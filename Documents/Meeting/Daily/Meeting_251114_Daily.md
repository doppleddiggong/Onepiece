## 2. 논의 요약

### 2.1. 주요 논의
- OT와 작업문화 규칙을 정리하여 신규 멤버는 반드시 작업 흐름을 파악한 뒤 투입하고, 야간·주말 작업은 화면 공유 등으로 투명하게 운영하기로 함.
- “입국/호텔/음식 주문 중 하나”를 중심으로 읽기·쓰기·말하기 스테이지를 통합한 단일 시나리오를 구성하고, 오늘 오전까지 확정한 흐름에 따라 Figma Flow와 PPT 제작을 이어가기로 함.
- Base64 통일 데이터 규약을 적용해 이미지·음성·텍스트를 전송하며, WebSocket 처리와 API 페이로드를 Base64 단위로 설계하기로 결정.

### 2.2. 기술 / 기획 관련 논의
- Whisper + 로컬 한국어 LLM, VectorDB 기반 RAG, 웹서치 역할 분리 등 AI 기술 범위를 정리하고, STT/TTS 로그 흐름과 발음 교정 피드백 시스템을 점검함.
- 음성/이미지 데이터는 주기적으로 서버에 전송하며 API는 인코딩된 페이로드 중심으로 구성, Base64 샘플을 빠르게 제작해 통신 안정성을 검증하는 시도를 병행.
- 피그마 Flow→PPT→월요일 리뷰→화요일 발표로 이어지는 문서화 프로세스를 세우고, 주말 동안 발표 자료·시나리오 정리를 완료할 계획.

### 2.3. 일정 / 운영 관련 논의
- 오늘 오전에 시나리오 1개를 확정하고, 오후부터 읽기·쓰기·말하기 스테이지 구성 요소와 피그마 흐름을 작성하여 주말 비주얼 작업을 뒷받침.
- WebSocket 기반 음성 처리 테스트와 Base64 API 샘플 제작을 병행하며, 월요일에 기술 설계 초안을 리뷰하고 화요일 발표 준비를 완료.
- DevOps(커밋 메시지 자동화, 문서/DevLog 파이프라인)도 오늘 중 적용하여 발표 자료 작성 시 협업 유실을 예방.

## 4. 회고

### 4.1. 잘된 점 (Keep)
- 읽기·쓰기·말하기 흐름을 하나의 상황에 통합한 시나리오 계획이 명확해졌고, OT에 대한 전원 합의로 작업 문화 안정화를 도모함.
- Base64 통신 규약과 AI 기술 범위(STT/TTS/VectorDB/웹서치)가 정리되어 서버/클라이언트 간 충돌을 줄일 준비가 된 상태.

### 4.2. 미진한 점 (Problem)
- 법률 기반 VectorDB 포함 범위와 웹서치/AI 교환 경계가 아직 팀 전체에 완전히 전파되지 않았음.
- 전체 Flow 문서와 발표 자료가 아직 통합되지 않아 발표 준비 시 문서량 조절이 필요함.

### 4.3. 다음 단계 (Try)
- 오전에 시나리오 확정과 Flow 작성 후 DevOps 적용을 병행하고, 주말 동안 PPT·설계 문서를 정리하여 월요일 리뷰에서 피드백을 반영.
- Base64 API 샘플 및 Whisper 테스트 결과를 정리해 월요일 기술 설계 초안에 포함하고, 화요일 발표 대비 문서 압축본을 함께 준비.
